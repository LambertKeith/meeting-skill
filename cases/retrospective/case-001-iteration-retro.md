# 案例：迭代复盘会

## 案例信息

| 项目 | 内容 |
|------|------|
| 案例编号 | case-001-iteration-retro |
| 会议类型 | 复盘会 |
| 使用方法 | 敏捷回顾（Stop/Start/Continue） |
| 参会Agent数 | 4 |
| 环节数 | 3 |
| 总时长 | 30分钟 |
| 效果评分 | 4.5 |

---

## 用户触发

```
"复盘这次迭代"
```

---

## 预期流程

### Phase 1: 收集反馈（10分钟）

**TeamLead广播**:
```
SendMessage({
  to: "*",
  summary: "敏捷回顾：收集反馈",
  message: "【敏捷回顾 Phase 1】收集反馈\n\n迭代：Sprint 2026-W14\n\n任务：按Stop/Start/Continue分类反馈\n时间盒：10分钟\n\n规则：具体、可操作，不空泛\n\n输出格式要求：\n## {你的Agent名称} - 敏捷回顾\n### Stop（停止）\n- ...\n### Start（开始）\n- ...\n### Continue（继续）\n- ...\n\n请在10分钟内完成输出。"
})
```

**Participant输出示例**:
```markdown
## participant-1 - 敏捷回顾

### Stop（停止）
- 停止在站会上讨论技术细节，导致站会超时
- 停止在需求澄清不完整时就开始开发

### Start（开始）
- 开始在需求评审时邀请测试人员参与
- 开始使用看板工具跟踪任务进度

### Continue（继续）
- 继续每日站会的节奏
- 继续代码Review机制

### 最优先改进项
- 需求评审引入测试人员参与
```

### Phase 2: 汇总分析（10分钟）

**TeamLead汇总**:
```markdown
## 敏捷回顾 Phase 1 汇总

### Stop高频项
| 项目 | 提及次数 | 建议 |
|------|----------|------|
| 站会超时 | 3 | 控制站会时间在15分钟内 |
| 需求澄清不完整 | 2 | 增加需求预审环节 |

### Start高频项
| 项目 | 提及次数 | 建议 |
|------|----------|------|
| 需求评审邀请测试 | 3 | 下迭代开始执行 |
| 使用看板工具 | 2 | 选择工具并培训 |

### Continue高频项
| 项目 | 提及次数 | 原因 |
|------|----------|------|
| 每日站会 | 4 | 保持节奏 |
| 代码Review | 3 | 保证质量 |

### 改进优先级排序
| 序号 | 改进 | 类型 | 优先级 |
|------|------|------|--------|
| 1 | 需求评审邀请测试人员 | Start | P1 |
| 2 | 控制站会时间 | Stop | P1 |
| 3 | 使用看板工具 | Start | P2 |
```

### Phase 3: 行动项制定（10分钟）

**TeamLead输出**:
```markdown
## 下迭代行动项（W3格式）

| Who | What | When | 类型 |
|-----|------|------|------|
| 产品经理 | 需求评审邀请测试人员参与 | Sprint W15开始 | Start |
| Scrum Master | 控制站会时间在15分钟内 | 每日执行 | Stop |
| 技术负责人 | 评估看板工具选择 | 4月15日前 | Start |

### DecisionMaker确认
- 确认人：decision-maker
- 确认时间：2026-04-08 15:30
- 确认行动项：以上3项行动项确认执行
```

## 效果评价

### 目标达成度
- ✅ 收集到充分的反馈
- ✅ 形成优先级排序
- ✅ 行动项具体可执行

### 会议亮点
- Stop/Start/Continue结构清晰
- 高频项统计帮助识别重点
- 行动项数量控制在3个

### 改进建议
- 可增加投票环节确认优先级
- 可增加上迭代行动项检查
