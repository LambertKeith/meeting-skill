# 会议记录演化机制

## 概述

会议记录不是静态文档，而是可演化、可复用的知识资产。本文档定义会议记录的生命周期管理和演化机制。

---

## 记录生命周期

| 阶段 | 说明 | 操作 |
|------|------|------|
| 创建 | 会议结束生成 | 自动生成标准结构 |
| 存储 | 按时间戳归档 | meeting-records/session-{timestamp}/ |
| 查询 | 按条件检索 | 类型、日期、关键词 |
| 复用 | 作为下次会议输入 | 引用历史决议、行动项 |
| 演化 | 版本迭代更新 | 新会议覆盖旧决议 |

---

## 记录目录结构

```
meeting-records/
├── session-20260408-001/
│   ├── v1/                      # 初始版本
│   │   ├── agenda.md
│   │   ├── roles.md
│   │   ├── phases/
│   │   ├── minutes.md
│   │   └── action-items.md
│   ├── v2/                      # 修正版本（如有修正）
│   │   ├── minutes.md
│   │   ├── correction-log.md    # 修正记录
│   │   └── action-items.md
│   ├── current/                 # 当前版本（软链接）
│   └── metadata.json            # 元数据
├── session-20260409-001/
└── index.md                     # 会议索引文件
```

## 版本管理机制

### 版本命名规则

| 版本 | 说明 |
|------|------|
| v1 | 初始版本，会议结束时生成 |
| v2, v3... | 修正版本，每次修正生成新版本 |
| current | 当前有效版本的软链接 |

### 版本切换

```markdown
## 版本切换操作

# 切换到v2版本
ln -sfn session-20260408-001/v2 session-20260408-001/current

# 版本切换记录
| 时间 | 从版本 | 到版本 | 操作者 | 原因 |
|------|--------|--------|--------|------|
| 2026-04-08 16:00 | v1 | v2 | TeamLead | 修正纪要错误 |
```

## 修正流程

### 触发条件

用户说：
- "修正刚才的纪要"
- "纪要里{某点}不准确"
- "行动项需要调整"

### 修正流程

```
用户请求修正
    ↓
TeamLead确认修正内容
    ↓
创建新版本目录(v{n+1})
    ↓
复制当前版本文件
    ↓
修改需要修正的内容
    ↓
写入correction-log.md
    ↓
更新current软链接
    ↓
通知相关方
```

### correction-log.md 格式

```markdown
# 修正记录

## 修正信息
| 项目 | 内容 |
|------|------|
| 修正版本 | v2 |
| 修正时间 | 2026-04-08 16:00 |
| 修正者 | TeamLead |
| 来源版本 | v1 |

## 修正内容

### 修正1
- **修正项**: {修正的文件和位置}
- **原内容**: {修正前的内容}
- **新内容**: {修正后的内容}
- **修正原因**: {用户请求的原因}

### 修正2
- **修正项**: {修正的文件和位置}
- **原内容**: {修正前的内容}
- **新内容**: {修正后的内容}
- **修正原因**: {用户请求的原因}

## 修正汇总
- 修正项数量：{N}
- 影响文件：{文件列表}
- 是否影响行动项：是/否
```

## 追加合并机制

### 触发条件

- 会议后有新增信息
- 行动项状态更新
- 决议补充说明

### 追加流程

```
追加信息
    ↓
分析增量内容
    ↓
确定追加位置
    ↓
合并入相关章节（不覆盖已有结论）
    ↓
标记追加内容
    ↓
更新metadata.json
```

### 追加标记

```markdown
## 原有内容
...

## 追加内容（2026-04-09 10:00）
> 以下内容为会后追加

- 追加项1: {内容}
- 追加项2: {内容}
```

## 记录复用方式

### 引用历史决议

在新会议的agenda.md中引用历史会议：

```markdown
## 相关历史会议
- 上次会议决议：参考 session-20260408-001/current/minutes.md#决议事项
- 待决事项：参考 session-20260408-001/current/minutes.md#待决事项
- 行动项进度：参考 session-20260408-001/current/action-items.md
```

### 行动项追踪

在新会议的action-items.md中关联历史行动项：

```markdown
## 行动项追踪

| 当前行动项 | 来源会议 | 历史状态 | 当前状态 |
|------------|----------|----------|----------|
| 更新流程文档 | session-20260408-001 | 待执行 | 已完成 |
| 建立变更流程 | session-20260408-001 | 待执行 | 进行中 |

### 上迭代行动项检查
| 行动项 | 来源 | 执行结果 |
|--------|------|----------|
| 需求评审邀请测试 | session-20260407-001 | 已执行，效果良好 |
```

## 演化规则

### 决议更新

当新会议覆盖旧决议时：

1. 在minutes.md中标注：
```markdown
## 决议更新记录
- 本会议决议覆盖：session-20260408-001（原决议：{内容}）
- 更新后决议：{新内容}
- 更新原因：{原因}
```

2. 在索引中更新决议状态

3. 保留历史记录，仅标记状态

### 行动项生命周期

| 状态 | 说明 | 触发条件 |
|------|------|----------|
| 待执行 | 新创建 | 会议结束 |
| 进行中 | 开始执行 | 执行者报告 |
| 已完成 | 完成执行 | 执行者确认 |
| 已取消 | 不再执行 | 新会议决议 |
| 已延期 | 超期未执行 | 截止日期后 |

### 行动项状态更新

```markdown
## 行动项状态更新

| 行动项 | 原状态 | 新状态 | 更新时间 | 更新原因 |
|--------|--------|--------|----------|----------|
| 建立变更流程 | 待执行 | 进行中 | 2026-04-10 | 开始执行 |
| 评估看板工具 | 待执行 | 已取消 | 2026-04-10 | 新会议决议暂停 |
```

## 清理机制

### 清理规则

| 规则 | 说明 |
|------|------|
| 保留期限 | 所有会议记录保留，不自动删除 |
| 归档条件 | 超过6个月的会议记录可归档压缩 |
| 清理条件 | 仅用户明确请求时清理 |

### 归档操作

```
session-20260408-001/
├── archive.zip          # 归档压缩包
└── metadata.json        # 保留元数据
```
