# 无责复盘 (Blameless Retrospective)

## 方法概述

特别适合事故、故障、线上问题等敏感场景。目标不是找替罪羊，而是理解系统为何允许问题发生。

## 核心原则

| 原则 | 说明 |
|------|------|
| 不追责个人 | 目标是理解系统，不是惩罚人 |
| 理解系统 | 为何系统允许问题发生 |
| 鼓励暴露 | 让大家敢于说真实原因 |
| 改进机制 | 改进系统而非追究个人 |

## Agent适配流程

### Phase 1 - 事件陈述

**TeamLead广播**：
```
【无责复盘】事件：{事件名称}
规则：
- 不追责个人
- 理解系统为何允许问题发生
- 鼓励暴露真实原因
- 改进机制而非人
```

**Contributor输出模板**：
```markdown
## {Agent名称} - 事件陈述

### 事件时间线
| 时间 | 事件 | 影响 |
|------|------|------|
| ... | ... | ... |

### 事件影响
- 用户影响: ...
- 业务影响: ...
- 系统影响: ...

### 我当时观察到的
- ...
```

### Phase 2 - 系统分析

**关键问题：系统为何允许问题发生？**

**Contributor输出模板**：
```markdown
## {Agent名称} - 系统分析

### 系统允许的原因（不是人的原因）

| 系统/机制 | 为何允许 | 改进方向 |
|-----------|----------|----------|
| 监控机制 | 未覆盖该指标 | 增加监控 |
| 告警机制 | 告警阈值不当 | 调整阈值 |
| 预案机制 | 未有该预案 | 补充预案 |
| 权限机制 | 权限过大 | 收缩权限 |
| 测试机制 | 未覆盖该场景 | 补充测试 |
| ...

### 根因归类
- 类型: [监控/告警/预案/权限/测试/流程/其他]
- 说明: {系统层面的原因}
```

### Phase 3 - 改进方案

**Contributor输出模板**：
```markdown
## {Agent名称} - 改进方案

### 改进项
| 序号 | 改进 | 针对系统 | 预期效果 |
|------|------|----------|----------|
| 1 | ... | 监控机制 | ... |
| 2 | ... | 告警机制 | ... |

### 防止同类问题
- 系统改进: ...
- 不依赖"人更细心"
```

### Phase 4 - 汇总确认

**TeamLead汇总**：
```markdown
## 无责复盘汇总

### 系统层面根因
| 根因 | 提及次数 | 改进方案 |
|------|----------|----------|
| 监控缺失 | {N} | ... |
| 告警阈值不当 | {N} | ... |

### 改进行动项
| 序号 | Who | What | When | 针对系统 |
|------|-----|------|------|----------|
| 1 | ... | ... | ... | 监控 |
| 2 | ... | ... | ... | 告警 |

### 强调
- 本复盘不追责个人
- 改进的是系统机制
- 目标是防止同类问题再次发生

### Approver确认
- 系统根因认可: ✅
- 改进方案批准: ✅
```

## 适用场景

- 事故复盘
- 故障复盘
- 线上问题
- 协作失误
- 任何敏感场景

## 注意事项

- 禁止"你应该..."的指责表述
- 用"系统允许..."替代"某人导致..."
- 改进必须针对机制层面
- 鼓励暴露真实问题，不隐瞒
- 对技术团队尤其重要：追责会让人隐藏问题
