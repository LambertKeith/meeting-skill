# 5Why 分析法

## 方法概述

连续追问"为什么"，追到深层根因。将问题从"某个人没做好"转向"机制缺口"。

## Agent适配流程

**TeamLead广播**：
```
【环节】5Why根因分析
任务：对问题{问题描述}进行连续追问
输出格式：完整的Why链条，直到根因
```

**Participant输出模板**：
```markdown
## {Agent名称} - 5Why分析

### 问题陈述
{原始问题}

### Why链条

**Why 1**: 为什么{问题}？
- 原因: ...

**Why 2**: 为什么{原因1}？
- 原因: ...

**Why 3**: 为什么{原因2}？
- 原因: ...

**Why 4**: 为什么{原因3}？
- 原因: ...

**Why 5**: 为什么{原因4}？
- 原因: ...

### 根因结论
- 根因类型: [人员/流程/工具/管理/其他]
- 根因描述: ...

### 对策建议
- ...
```

## 示例

```
问题：项目延期了

Why 1: 为什么延期？ → 测试晚了
Why 2: 为什么测试晚了？ → 开发提测延迟
Why 3: 为什么开发提测延迟？ → 需求变更多
Why 4: 为什么需求变更多？ → 前期需求澄清不足
Why 5: 为什么澄清不足？ → 关键业务方没有在方案评审前参与

根因：机制缺口 - 业务方参与时机不对
对策：方案评审前必须包含业务方确认环节
```

## 适用场景

- 故障分析
- 项目延期复盘
- 质量问题调查
- 流程失效分析

## 注意事项

- 不是所有问题都能线性追问，有些是多因素并发
- 追问要有依据，不能瞎猜
- 根因应该是可改进的机制层面，而非个人层面
