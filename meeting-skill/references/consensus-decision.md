# 共识决策法 (Consensus Decision-Making)

## 方法概述

通过充分讨论，形成大家都能接受的方案。不是"全员同意"，而是"无人强烈反对"。

## 核心原则

| 原则 | 说明 |
|------|------|
| 充分讨论 | 确保各方表达 |
| 理解差异 | 理解不同立场原因 |
| 寻找共同点 | 找出可接受的方案 |
| 设置边界 | 不追求完美一致 |

## Agent适配流程

### Phase 1 - 方案提议

**TeamLead广播**：
```
【共识决策】议题：{议题}
任务：提出可能的方案
规则：先发散，后收敛
```

**Contributor输出模板**：
```markdown
## {Agent名称} - 方案提议

### 方案
- 内容: ...

### 优势
- ...

### 劣势
- ...

### 我的立场
- [完全支持 / 基本支持 / 可接受 / 有顾虑 / 反对]
```

### Phase 2 - 理解差异

**TeamLead广播**：
```
【共识决策】理解差异
任务：分析各方案的支持/反对理由
规则：理解立场，不争论
```

**Contributor输出模板**：
```markdown
## {Agent名称} - 理解差异

### 我对方案{A}的看法
- 支持/顾虑/反对的原因: ...

### 我对方案{B}的看法
- ...

### 我能接受的方案
- {方案X}，原因: ...

### 我不能接受的方案
- {方案Y}，原因: ...
```

### Phase 3 - 寻找共识

**TeamLead分析汇总**：
```markdown
## 共识分析

### 各方案接受度
| 方案 | 支持 | 可接受 | 有顾虑 | 反对 |
|------|------|--------|--------|------|
| A | {N} | {N} | {N} | {N} |
| B | {N} | {N} | {N} | {N} |

### 反对/顾虑原因
| 方案 | 反对者 | 原因 | 可否调整 |
|------|--------|------|----------|
| A | Agent X | ... | {调整建议} |

### 可能共识方向
- {方案} + {调整} 可能让更多人接受
```

### Phase 4 - 调整并确认

**TeamLead提出调整方案**：
```
【共识决策】调整方案
原方案：{方案}
调整：{调整内容}
请确认是否可接受。
```

**Contributor确认**：
```markdown
## {Agent名称} - 最终确认

### 调整后方案：{方案}
### 我的立场
- [可接受 / 仍有顾虑 / 反对]

### 如有顾虑/反对
- 原因: ...
- 是否阻断执行: [是/否]
```

### Phase 5 - 决议

**TeamLead汇总**：
```markdown
## 共识决策结果

### 最终方案
{方案内容}

### 接受情况
| Agent | 立场 | 备注 |
|-------|------|------|
| A | 可接受 | |
| B | 可接受 | 有轻微顾虑但不阻断 |
| C | 反对 | 原因...，但同意不阻断执行 |

### 决议
- 达成共识: ✅
- 阻断执行: ❌
- Approver确认: {签名}
```

## 适用场景

- 高协作、高互赖团队
- 如果有人强烈反对，后续执行会受阻
- 跨部门协作
- 重要但非紧急决策

## 注意事项

- 速度慢，需充分时间
- 需设置边界：不追求完美一致
- 防止少数强反对者拖住
- Driver需控制节奏，适时推进
- "可接受共识"≠"全员同意"
