# 鱼骨图分析法 (Fishbone / Ishikawa)

## 方法概述

多因素问题分析，把原因按类别归纳，适合质量管理、故障分析、运营异常等。

## 分析维度

常见维度（可根据问题调整）：

| 维度 | 含义 | 典型问题 |
|------|------|----------|
| 人 | 人员因素 | 培训不足、经验不够、责任心 |
| 流程 | 流程因素 | 响应机制、升级路径、审批流程 |
| 工具 | 工具因素 | 系统延迟、信息不同步、工具缺陷 |
| 数据 | 数据因素 | 数据缺失、数据质量、数据延迟 |
| 环境 | 环境因素 | 物理环境、网络环境、外部依赖 |
| 管理 | 管理因素 | 指标导向、资源配置、激励机制 |

## Agent适配流程

### Phase 1 - 定义问题

**输出模板**：
```markdown
## 问题定义

### 问题陈述
{要分析的问题}

### 问题现象
- ...
- ...

### 问题影响
- ...
```

### Phase 2 - 分维度分析

**TeamLead广播**：
```
【鱼骨图分析】分维度分析原因
任务：从各维度分析可能的因素
维度：人、流程、工具、数据、环境、管理
```

**Contributor输出模板**：
```markdown
## {Agent名称} - 鱼骨图分析

### 问题：{问题陈述}

| 维度 | 可能因素 | 证据/依据 |
|------|----------|-----------|
| 人 | 1. ...<br>2. ... | ... |
| 流程 | 1. ...<br>2. ... | ... |
| 工具 | 1. ...<br>2. ... | ... |
| 数据 | 1. ...<br>2. ... | ... |
| 环境 | 1. ...<br>2. ... | ... |
| 管理 | 1. ...<br>2. ... | ... |

### 最可能因素
- 维度: {维度}
- 因素: {具体因素}
- 理由: {依据}
```

### Phase 3 - 汇总验证

**TeamLead汇总**：
```markdown
## 鱼骨图汇总

### 高频因素（多Agent提及）
| 因素 | 提及次数 | 维度 | 验证建议 |
|------|----------|------|----------|
| ... | {N} | ... | {如何验证} |

### 需验证因素
- ...

### 分析结论
- 主要原因方向: {维度}
- 待验证项: {具体因素}
```

## 适用场景

- 故障分析
- 质量问题调查
- 客户投诉分析
- 运营异常调查

## 注意事项

- 因素需要后续验证，不能只停留在"列原因"
- 多因素并发时，鱼骨图优于5Why
- 维度可根据问题类型调整（如技术问题可用：代码、架构、依赖、配置、环境、运维）
